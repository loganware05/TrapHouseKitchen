# Environment Variables for Render Deployment

## After deploy: dashboard checklist

Do these in the Render Dashboard once `traphousekitchen-api`, `traphousekitchen-web`, and `traphousekitchen-db` exist.

| Variable | Service | Value / notes |
| -------- | ------- | ------------- |
| `FRONTEND_URL` | `traphousekitchen-api` | `https://<your-static-host>` — exact `Origin` the browser sends (e.g. `https://traphousekitchen-web.onrender.com`). **No trailing slash.** |
| `VITE_API_URL` | `traphousekitchen-web` | `https://<your-api-host>/api` — public HTTPS API base **including `/api`**. |
| `VITE_STRIPE_PUBLISHABLE_KEY` | `traphousekitchen-web` | Same mode as backend (`pk_test_...` vs `pk_live_...`). |
| Stripe, email, chef, AI secrets | `traphousekitchen-api` | See sections below. |

**Important:** `VITE_*` variables are embedded at **build** time. After you change `VITE_API_URL` or `VITE_STRIPE_PUBLISHABLE_KEY`, trigger a **new deploy** of the static site (Clear build cache + deploy if needed) so the bundle picks them up.

---

## Backend Service (`traphousekitchen-api`)

Copy or confirm these in Render Dashboard → Environment:

```bash
# Database — supplied by blueprint (fromDatabase) for fresh installs using repo render.yaml
# No manual paste needed unless you use an external database instead of traphousekitchen-db

# JWT
JWT_SECRET=<auto-generated by Render>
JWT_EXPIRES_IN=7d

# CORS — you must set manually (Render cannot derive this from YAML)
FRONTEND_URL=https://traphousekitchen-web.onrender.com

# Stripe
STRIPE_SECRET_KEY=sk_live_xxxxxxxxxxxxx
STRIPE_PUBLISHABLE_KEY=pk_live_xxxxxxxxxxxxx
STRIPE_WEBHOOK_SECRET=whsec_xxxxxxxxxxxxx

# Email (Resend)
RESEND_API_KEY=re_xxxxxxxxxxxxx
FROM_EMAIL=TrapHouse Kitchen <orders@traphousekitchen.com>
CHEF_EMAIL=chef@traphousekitchen.com

# Environment
NODE_ENV=production
PORT=10000
```

## Frontend Static Site (`traphousekitchen-web`)

```bash
VITE_API_URL=https://traphousekitchen-api.onrender.com/api
VITE_STRIPE_PUBLISHABLE_KEY=pk_live_xxxxxxxxxxxxx
```

## Quick setup guide

### 1. Backend

**From blueprint:** `DATABASE_URL` is wired from `traphousekitchen-db` via `fromDatabase` in `render.yaml`.

**You set in dashboard:** `FRONTEND_URL`, Stripe keys, webhook secret, `CHEF_EMAIL`, optional Resend, optional AI keys — see [render.md](./render.md).

**Auto-generated:** `JWT_SECRET` (if using blueprint defaults).

**Preset in YAML:** `NODE_ENV`, `PORT`, `JWT_EXPIRES_IN`, `FROM_EMAIL`, etc.

### 2. Frontend

**Required:** `VITE_API_URL` (backend public URL + `/api`), `VITE_STRIPE_PUBLISHABLE_KEY`.

**Redeploy** the static service whenever `VITE_*` changes.

---

## Important notes

1. **Order:** Database and API should be live before the static site build that needs a real `VITE_API_URL` (preview URLs differ — see pull request / CORS notes in [render.md](./render.md)).
2. **`FRONTEND_URL`:** Must match the deployed frontend origin exactly or CORS will block the API (common on iOS Safari).
3. **Stripe keys:** Use the same mode (test vs live) on API and static site.
4. **Webhook secret:** Create the Stripe webhook after the API URL is known.
5. **External Postgres:** If you ever attach a non-Render database, set `DATABASE_URL` in the dashboard and remove or override the blueprint `fromDatabase` entry for that environment.

## How to get each value

### DATABASE_URL

- **Default (this repo):** Provided automatically from `traphousekitchen-db` by the blueprint.
- **Manual / external:** Render Dashboard → PostgreSQL → Connect → copy URL; include `?sslmode=require` if required by your client.

### Stripe keys & webhook

Same as before: Stripe Dashboard → Developers; webhook URL `https://<api-host>/api/webhooks/stripe` (adjust path if your app differs).

### FRONTEND_URL and VITE_API_URL

1. Copy the static site URL from Render (no trailing slash for `FRONTEND_URL`).
2. Copy the API service URL; set `VITE_API_URL` to that origin plus `/api`.
3. Redeploy the static site after setting or changing `VITE_*`.
